# 七張 Sheet 原子快照同步設計

日期：2026-07-13

狀態：Bar 已核准書面規格與單一原子狀態包修正

目標分支：`dev`

## 1. 背景與問題

App 目前會平行下載七張 Google Sheet。每張下載成功後便立即覆寫該 Sheet 的 `RAW`、來源狀態與 `v2_cache_*` localStorage；所有工作結束後再重建 `DB`。若只有部分 Sheet 成功，執行期與下次啟動都可能同時使用不同同步批次的資料。

這會讓跨 Sheet 關聯失去版本一致性，例如行程卡的 Restaurant Ref 與 Restaurants 內容不一致。使用者看到「部分更新」時，實際上可能已經使用混合版本資料。

本設計把七張 Sheet 視為一個不可分割的資料版本：只有下載、解析、驗證與持久化全部成功，候選版本才可一次生效。

## 2. 目標

- 七張 Sheet 全部成功且通過驗證後才切換資料。
- 任一階段失敗時，正在顯示與已保存的完整版本保持不變。
- 同步驗證與診斷面板使用同一套驗證核心。
- 相容現有七份 `v2_cache_*`，以漸進方式遷移。
- 保留一份上一個完整快照供損壞恢復。
- 使用者能知道目前資料來源、最後完整同步時間與最近失敗原因。
- 維持 ES5/iOS WebView 相容性與目前 localStorage 架構。

## 3. 非目標

- 不新增或修改 Google Sheet 欄位。
- 不導入 IndexedDB、帳號、資料庫或後端服務。
- 不修改分帳頁功能。
- 不混入無障礙、天氣、Day 標籤或其他 UX 改善。
- 不在本批完成完整 Playwright 多尺寸測試；瀏覽器整合測試維持獨立批次。
- 不修改各頁 renderer 的資料呈現規則，只有資料版本切換與同步狀態 UI 例外。

## 4. 採用方案與替代方案

### 4.1 採用：localStorage 完整快照加兩階段切換

七張下載結果先停留在記憶體候選區。候選資料通過所有阻擋驗證後，整包寫入一個完整快照；持久化成功後，才一次切換執行期 `RAW`、`SRC` 與 `DB` 並重繪。

此方案沿用現有儲存技術、變更範圍集中，最符合目前資料量與 iOS 相容要求。

### 4.2 未採用：七份版本化快取加 active generation

每張 Sheet 分別儲存並帶同一 generation ID，最後更新 active generation。此方案可細分單張資料，但 key 清理、回退與孤兒版本管理較複雜，沒有足夠收益。

### 4.3 未採用：IndexedDB transaction

IndexedDB 有較佳容量與交易能力，但對目前小型單人 PWA 過度複雜，且會增加 iOS 儲存異常、migration 與測試成本。

## 5. 快照資料模型

完整快照至少包含：

- `formatVersion`：快照格式版本。
- `generationId`：此批同步的唯一識別。
- `createdAt`：完整同步成功時間。
- `source`：`online`、`legacy-migrated` 或 `builtin`。
- `sheets`：七張 Sheet 的完整 CSV 字串。
- `sheetMeta`：各 Sheet 的擷取結果與摘要，不保存診斷用完整副本。
- `validation`：通過時的警告摘要與驗證器版本。

localStorage 使用一個原子狀態包與一個輕量失敗摘要：

- `trip_data_snapshot_state`：物件內含 `active` 與 `previous`；兩者透過一次 `setItem()` 同時替換。
- `trip_sync_last_failure`：最近一次失敗摘要，不含完整 CSV。

第一版 `formatVersion` 固定為 `1`。七張 Sheet 不得分別成為可獨立啟用的正式版本。

`trip_data_snapshot_state` 的外層結構為：

```text
{
  formatVersion: 1,
  active: { ...目前完整快照... },
  previous: { ...上一個完整快照或 null... }
}
```

localStorage 不提供跨 key transaction，因此不得將 active 與 previous 分成兩個正式 key。單一狀態包確保寫入失敗時，舊 active 與 previous 會一起保留。

## 6. 啟動與遷移

啟動時依序嘗試：

1. 讀取並解析 `trip_data_snapshot_state`。
2. 驗證狀態包內的 active；有效則使用。
3. active 損壞時驗證同一狀態包內的 previous；有效則回退使用。
4. active 與 previous 都無效或尚不存在時，讀取現有七份 `v2_cache_*`。
5. 舊快取只有在七份都存在，且組合後通過完整阻擋驗證時，才包裝成 `legacy-migrated` snapshot。
6. 舊快取不完整或驗證失敗時，改用通過內建驗證的 BUILTIN。
7. 畫面可用後，背景嘗試取得最新線上完整版本。

第一次成功建立新格式狀態包後，舊 `v2_cache_*` 暫留作為遷移期保護，但不再被更新，也不再優先於新格式快照。後續清理舊格式必須是獨立、可失敗且不影響 active 的操作。

## 7. 線上同步與兩階段切換

### 7.1 候選階段

1. 建立單一同步工作識別，阻止重複同步並行。
2. 平行下載七張 Sheet。
3. 結果只寫入記憶體候選集合，不修改正式 `RAW`、`SRC`、`DB` 或 localStorage。
4. 對候選集合執行 CSV 解析、Schema 驗證、DB 建立與跨 Sheet 關聯驗證。

### 7.2 提交階段

候選資料沒有阻擋錯誤時：

1. 建立完整候選快照物件。
2. 建立下一個狀態包：候選快照為 `active`，目前有效 active 為 `previous`。
3. 序列化整個狀態包並以一次 `setItem('trip_data_snapshot_state', ...)` 寫入。
4. 確認整個狀態包可讀回、解析且 active generation 正確。
5. 才將執行期 `RAW`、`SRC`、`DB` 一次切換到新版本。
6. 全 App 只重繪一次。
7. 更新同步狀態與最後完整同步時間。

若 localStorage 寫入或讀回確認失敗，不得只在記憶體啟用候選資料，避免重新啟動後版本倒退或不一致。

### 7.3 失敗不變性

任一下載、解析、驗證、DB 建立或持久化失敗時：

- `trip_data_snapshot_state` 不變，因此 active 與 previous 同時保持不變。
- 正式 `RAW`、`SRC` 與 `DB` 不變。
- 不留下可被誤判為正式版本的部分快取。
- 記錄失敗摘要並更新非阻斷式狀態 UI。

## 8. 分級驗證

同步驗證必須接受候選 RAW/DB 作為輸入，不得透過暫時覆寫全域正式資料來執行。現有 `healthCheck()` 應共用同一個可測試的驗證核心，使同步 Gate 與診斷面板不會產生不同結論。

### 8.1 阻擋整批更新

- 任一 Sheet 下載失敗、逾時、HTTP 錯誤或內容為空。
- CSV 無法解析。
- 找不到必要標題或必要欄位。
- 重複 PID、RID、Shopping ID 或其他主鍵。
- 行程 Ref 指向不存在的 Places 或 Restaurants。
- Shopping 或 Restaurants 指向不存在的 PID。
- 行程顯示名稱與 Ref 對應資料明顯不一致。
- 未定義的 Places Type。
- TripConfig 缺少系統運作所需的必要設定。
- 候選資料無法建立完整 DB。
- 新快照無法完整寫入或讀回 localStorage。

### 8.2 只警告、不阻擋

- 選填欄位未填，例如官網、門票、停車或備註。
- 無人引用的資料列。
- 無法辨識但不影響正式欄位的額外欄位。
- 建議格式不一致，但仍能安全解析的內容。

### 8.3 錯誤紀錄

最近一次失敗摘要包含：

- 發生時間。
- 階段：下載、解析、驗證或儲存。
- Sheet 名稱。
- 穩定錯誤代碼與中文說明。
- 當時沿用的完整快照時間。

不在診斷紀錄中保存完整 CSV，避免 localStorage 與複製報告過大。

## 9. 同步狀態 UI

Header 只顯示適合手機寬度的短狀態：

- `同步中`
- `已是最新`
- `更新失敗`
- `離線版`
- `內建版`

點擊 Header 同步狀態後開啟「資料同步狀態」面板。原本直接同步的操作移到面板內的「立即重新同步」按鈕。

面板顯示：

- APP build。
- 目前 active generation 或可理解的資料版本。
- 資料來源。
- 最後完整同步時間。
- 最近一次失敗階段、Sheet 與原因。
- 非阻擋警告。
- 立即重新同步按鈕。

同步進行中按鈕必須 disabled，避免並行工作。同步成功後面板留在原處並更新內容，不強迫關閉。

首次自動同步失敗時，只顯示一次非阻斷提示：「更新失敗，已保留上一個完整版本。」完整訊息在面板中顯示，例如「更新失敗，正在沿用 2026/07/13 21:30 的完整版本」。

若沒有有效快照而使用 BUILTIN，Header 必須顯示「內建版」，不得顯示「已是最新」。

## 10. P025 Type 上線前置條件

2026-07-13 檢查公開 Places CSV 時，`P025 大步危峽谷觀光遊覽船` 的 Type 仍為未定義的「遊覽船」。Bar 已將 Sheet Type 改為 Schema 既有允許值「渡輪」，但 Google 公開 CSV 當下尚未刷新。

實作與推送前必須再次直接查核公開 CSV：P025 Type 已顯示「渡輪」。若仍為「遊覽船」，新的嚴格驗證應阻擋候選資料；不得為了讓同步通過而臨時放寬未知 Type 規則。

## 11. 測試與驗收

必要自動測試：

- 七張全成功且驗證通過時，只切換一次並只重繪一次。
- 任一張下載失敗時，正式 RAW、DB 與 `trip_data_snapshot_state` 完全不變。
- 阻擋錯誤不生效；只有警告仍可生效。
- 名稱與 Ref 不一致、重複 ID、失效關聯與未知 Type 均被阻擋。
- localStorage 寫入或讀回失敗時，不切換記憶體資料。
- 成功更新後，單一狀態包內的舊 active 成為 previous，且只保留一份 previous。
- active 損壞時可回退 previous。
- 七份舊快取只有通過完整驗證才可遷移。
- 無有效快取時使用 BUILTIN 並顯示「內建版」。
- 重複點擊同步不會建立兩個並行工作。
- 成功、失敗、離線、內建與警告狀態正確。
- 公開 CSV 的 P025 Type 必須為 Schema 已允許的「渡輪」。
- 現有 13 個測試與文件檢查維持通過。

本批手機驗收至少確認：

- 正常連線可完成整批更新。
- 暫時斷網時仍顯示上一個完整版本。
- 失敗面板清楚顯示沿用版本與重試入口。
- 重試成功後狀態與最後完整同步時間正確更新。
- 行程、購物、完成／略過與想逛資料呈現沒有回歸。

## 12. 風險與控制

- **localStorage 容量不足**：提交前序列化完整快照；寫入與讀回確認失敗時保持舊 active，不啟用候選版本。
- **舊快取本身已混版**：舊快取必須通過與線上候選相同的完整驗證；不通過便使用 BUILTIN。
- **驗證器誤判造成無法更新**：使用穩定錯誤碼與詳細面板；不可自動放寬 Gate，需依治理流程修正 Schema 或 Sheet。
- **同步期間重複操作**：單一 in-flight guard，UI 同步按鈕 disabled。
- **切換後 renderer 例外**：候選 DB 必須在提交前完整建立；正式切換與重繪集中在單一提交函式。
- **Google 發布延遲**：以公開 CSV 為準，不以 Sheet 編輯畫面已修改作為成功依據。

## 13. 成功標準

- App 永遠只使用一個可識別的七張 Sheet 完整版本。
- 部分下載成功不再造成新舊資料混用。
- 同步失敗後，重新開啟 App 仍使用同一個上一版完整快照。
- 使用者可以在正常 UI 理解目前資料版本、失敗原因與重試方式。
- R012/R013 類跨 Sheet 名稱與 Ref 不一致會在候選階段被阻擋，而不是進入正式畫面。
