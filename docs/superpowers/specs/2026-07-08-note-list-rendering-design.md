# 備註條列顯示優化設計

## 背景

Places、Restaurants、Hotels 與行程資料的 `備註` 欄位目前是自由文字。Sheet 中已經會用換行、`•`、`1.`、`①`、`※` 等格式整理多點資訊，但 App 顯示時多數位置仍以一般文字區塊呈現，手機上較難快速掃讀。

本次設計目標是改善備註的閱讀性，不改變資料來源與欄位結構。

## 範圍

- 僅改善 App 顯示層的備註呈現。
- 不修改 Google Sheet 欄位。
- 不修改 `schema.js` 欄位定義。
- 不修改 parser 資料結構。
- 不新增 CMS 欄位。
- 不改變現有備註內容的語意。

## 建議方案

新增共用備註渲染邏輯，例如 `renderNote()`，由 App 在顯示時自動辨識自由文字中的簡單條列格式。

支援格式：

- `• 文字`
- `- 文字`
- `1. 文字`
- `1、文字`
- `① 文字`
- `※ 文字`
- 一般換行段落

顯示規則：

- 一般句子顯示為段落。
- `•`、`-`、數字、圈號開頭的行顯示為清單項目。
- `※` 開頭的行顯示為提醒項目，可使用較醒目的樣式。
- 保留現有網址、TEL、票價等醒目處理。
- 空行用於分段，但不產生空白清單項。

## 套用位置

優先套用在以下已顯示備註的區塊：

- Places 詳細資訊備註。
- Restaurants 備註。
- Hotels 備註。
- 行程卡片中的備註。
- 購物地點卡片中的 Places 備註。

不套用在 Shopping 店家列的短備註，避免店家清單高度被放大。

## UI 原則

- 備註仍維持一個資訊區塊，不拆成新的資料欄位。
- 長內容以縮排清單提升可讀性。
- 行距比一般段落稍大，方便手機閱讀。
- 提醒項目用輕量強調，不做過重警示。
- 不增加新的操作按鈕，避免旅途中操作變複雜。

## 錯誤與相容處理

- 若備註沒有條列格式，顯示結果應接近目前樣式。
- 若使用者輸入混合格式，逐行判斷並保留原順序。
- 若備註含網址，仍可點擊。
- 若備註含特殊字元，仍需經過 HTML escape。

## 驗收條件

- Sheet 備註維持原格式即可被 App 顯示為較清楚的條列。
- `schema.js` 未修改。
- Google Sheet Schema 未修改。
- `09_SCHEMA_MAPPING.md` 未修改。
- 一般短備註不被過度格式化。
- 手機寬度下長備註不擠成一整段。
- 無 Breaking Change。
